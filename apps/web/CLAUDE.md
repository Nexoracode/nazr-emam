# قوانین فرانت — apps/web

> قوانین ریشه (`/CLAUDE.md`) را هم رعایت کن. این فایل فقط قوانین اختصاصی Next.js frontend است.

## مسئولیت این اپ

`apps/web` تجربه اصلی کاربر و مدیر را می‌سازد:

- صفحه اصلی و معرفی سامانه
- نمایش نوع‌های نذر و محتوای CMS
- فرم ثبت درخواست نذر
- ثبت پرداخت یا رسید
- صفحه پیگیری با کد رهگیری
- داشبورد کاربر
- پنل مدیریت درخواست‌ها، پرداخت‌ها و پیام‌ها

داده تراکنشی از `apps/api` می‌آید. محتوای قابل ویرایش، بنرها، متن‌های معرفی، FAQ و مدیا از `apps/cms` می‌آید.

## فارسی و RTL

- کل UI فارسی و `dir="rtl"` است.
- متن‌ها باید ساده، روشن و مناسب کاربر عمومی باشند.
- layout باید در موبایل و دسکتاپ سالم باشد.
- از CSS logical استفاده کن:
  - `padding-inline`
  - `margin-inline`
  - `inset-inline`
  - `border-inline`

## دیتا و API

- همه فراخوانی‌های NestJS API باید با قرارداد `docs/API-CONTRACT.md` هماهنگ باشد.
- تایپ داده‌ها را از `@nazr-emam/shared` بگیر.
- آدرس API از `NEXT_PUBLIC_API_URL` خوانده شود.
- آدرس CMS بعد از قطعی شدن ابزار CMS از env خوانده می‌شود؛ hard-code نکن.
- fetchهای مشترک را در یک لایه مثل `lib/api.ts` و بعداً `lib/cms.ts` متمرکز کن.

## کامپوننت و UI

- فرم‌ها باید `loading`، `error` و `success` state داشته باشند.
- خطاهای API باید برای کاربر قابل فهم نمایش داده شوند.
- مسیرهای اصلی پروژه ناقص رها نشوند:
  - ثبت نذر
  - پرداخت/رسید
  - پیگیری وضعیت
  - مدیریت درخواست‌ها
- طراحی باید آرام، قابل اعتماد و خدماتی باشد؛ از ظاهر شلوغ و تبلیغاتی پرهیز کن.

## Server و Client Component

- پیش‌فرض با Server Component کار کن.
- `"use client"` فقط وقتی لازم است: فرم تعاملی، state، event handler، modal یا upload.
- داده عمومی را تا حد امکان سمت سرور بگیر.

## چیزهایی که نباید بکنی

- تایپ response را داخل web دوباره ننویس.
- endpoint خارج از قرارداد را صدا نزن.
- متن UI را مبهم، طولانی یا تبلیغاتی ننویس.
- اطلاعات حساس کاربر را در صفحه عمومی نمایش نده.
- بدون درخواست کاربر i18n اضافه نکن.

## دستورها

```bash
npm run dev -w @nazr-emam/web
npm run build -w @nazr-emam/web
npm run lint -w @nazr-emam/web
```

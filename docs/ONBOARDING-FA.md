# راهنمای کامل پروژه — نذر امام

این سند را قبل از شروع هر کار بخوان. هدفش این است که بدانی پروژه چیست، چطور چیده شده، چطور اجرا می‌شود، و چطور باید تغییرات را جلو ببری.

---

## ۱) پروژه چیست؟

«نذر امام» یک سامانه فارسی و راست‌به‌چپ برای مدیریت نذرهاست. کاربر می‌تواند نذر یا درخواست خود را ثبت کند، اطلاعات تماس و جزئیات لازم را بدهد، پرداخت یا رسید پرداخت را ثبت کند و وضعیت انجام نذر را پیگیری کند. مدیر سامانه هم می‌تواند نذرها، درخواست‌ها، پرداخت‌ها، گزارش‌ها و پیام‌های کاربران را مدیریت کند.

پروژه در چند فاز ساخته می‌شود:

- **فاز ۱:** زیرساخت monorepo، Docker، health check، صفحه اولیه و قرارداد API.
- **فاز ۲:** ثبت نذر، نوع نذر، فرم‌های اصلی و پیگیری وضعیت.
- **فاز ۳:** پنل مدیریت، مدیریت پرداخت‌ها، گزارش‌گیری و تغییر وضعیت درخواست‌ها.
- **فاز ۴:** احراز هویت، داشبورد کاربر، اعلان‌ها، پیامک/ایمیل و امکانات تکمیلی.

---

## ۲) معماری: سه اپ که با هم یک پروژه‌اند

این پروژه یک **monorepo** است: یک پوشه واحد که فرانت، بک‌اند و CMS را کنار هم نگه می‌دارد.

```text
nazr-emam/
├── apps/
│   ├── web/      -> سایت و پنل‌ها با Next.js
│   ├── api/      -> بک‌اند و منطق کسب‌وکار با NestJS
│   └── cms/      -> مدیریت محتوای قابل ویرایش
├── packages/
│   └── shared/   -> تایپ‌های مشترک TypeScript بین فرانت و بک‌اند
├── docs/         -> مستندات، workflow و قرارداد API
├── Dockerfile    -> اجرای production هر دو اپ داخل یک container
├── docker-compose.yml
├── turbo.json    -> اجرای taskها با Turborepo
└── package.json  -> ریشه workspace با npm
```

### هر اپ چه‌کار می‌کند؟

| اپ | مسئولیت |
|---|---|
| `apps/web` | صفحه اصلی، فرم ثبت نذر، پیگیری وضعیت، داشبورد کاربر و پنل مدیریت |
| `apps/api` | ثبت و مدیریت درخواست‌ها، پرداخت‌ها، احراز هویت، گزارش‌ها و اعلان‌ها |
| `apps/cms` | مدیریت محتوای قابل ویرایش مثل بنرها، صفحه‌ها، FAQ، مدیا و متن‌های معرفی |

> مرز مهم: داده تراکنشی و امن از `apps/api` می‌آید؛ محتوای قابل ویرایش از `apps/cms`. هر داده‌ای که بین اپ‌ها رد و بدل می‌شود باید در `docs/API-CONTRACT.md` تعریف شده باشد و تایپ آن در `packages/shared` وجود داشته باشد.

## ۴) قرارداد API و تایپ مشترک

سند [`docs/API-CONTRACT.md`](./API-CONTRACT.md) منبع حقیقت پروژه است. قبل از ساخت هر endpoint، باید در این سند مشخص شود:

- مسیر endpoint چیست.
- چه ورودی می‌گیرد.
- چه خروجی برمی‌گرداند.
- چه خطاهایی دارد.
- عمومی است یا نیاز به لاگین/admin دارد.

قانون طلایی:

```text
اول API-CONTRACT.md و تایپ مشترک، بعد کد.
```

اگر نام یک فیلد، ساختار پاسخ، وضعیت‌ها یا کد خطا تغییر کرد، اول سند قرارداد و `packages/shared` را تغییر بده. بعد API و فرانت را با آن هماهنگ کن.

## ۵) راه‌اندازی روی سیستم خودت

### پیش‌نیازها

- Node.js نسخه ۲۲ یا نسخه LTS سازگار
- npm
- Git
- Docker Desktop برای تست production container

### نصب و اجرای local

```bash
npm install
npm run dev
```

بعد از اجرا:

- Frontend: `http://localhost:3000`
- API: `http://localhost:3001`
- Health check: `http://localhost:3001/health`

### build کل پروژه

```bash
npm run build
```

این دستور با Turbo، workspaceها را build می‌کند.

### اجرای Docker

```bash
docker compose up --build
```

قاعده پروژه این است که در production بخش‌های قابل اجرا داخل یک container اجرا شوند. فعلاً `api` و `web` اجرا می‌شوند؛ CMS بعد از انتخاب ابزار به همین قرارداد اضافه می‌شود.

## ۶) گردش کار روزانه

همه روی یک repo مشترک کار می‌کنند، ولی هر کار باید روی branch جدا انجام شود. مستقیم روی `main` کد نزن.

```text
۱. آخرین کد را بگیر:
   git pull
   npm install      <- اگر package.json یا package-lock.json تغییر کرده بود

۲. برای کارت branch بساز:
   git checkout -b feature/register-nazr-request

۳. قرارداد را بررسی کن:
   docs/API-CONTRACT.md

۴. کد بزن و local تست کن:
   npm run dev

۵. قبل از PR بررسی کن:
   npm run build
   npm run lint

۶. تغییرات را ارسال کن:
   git add -A
   git commit -m "Add nazr request registration"
   git push -u origin feature/register-nazr-request

۷. Pull Request باز کن و منتظر review بمان.
```

## ۷) قوانین کدنویسی

- **TypeScript strict:** تایپ مبهم نساز. شکل داده‌ها باید مشخص باشد.
- **قرارداد API:** هر endpoint باید با `API-CONTRACT.md` هماهنگ باشد.
- **تایپ مشترک:** تایپ داده‌های مشترک را در `@nazr-emam/shared` تعریف کن؛ تایپ تکراری محلی نساز.
- **RTL:** در CSS تا حد ممکن از خواص logical استفاده کن، مثل `padding-inline-start`.
- **متن فارسی:** متن‌های UI باید واضح، کوتاه و مناسب کاربر عمومی باشد.
- **رنگ و طراحی:** ظاهر باید ساده، قابل اعتماد و مناسب سامانه خدماتی/مذهبی باشد؛ نه شلوغ و تبلیغاتی.
- **Docker:** تغییری که production را خراب کند قابل قبول نیست. بعد از تغییرات زیرساختی، Docker هم باید تست شود.
- **Turbo:** taskها از ریشه با Turbo اجرا می‌شوند؛ برای اجرای کل پروژه از اسکریپت‌های root استفاده کن.

## پرسش‌های متداول

**آیا این پروژه CMS دارد؟**  
بله، workspace آن در `apps/cms` وجود دارد. ابزار CMS هنوز قطعی نشده، پس تا قبل از تصمیم نهایی فقط قرارداد و قوانین آن نگه‌داری می‌شود.

**چرا هر دو پروژه داخل یک container هستند؟**  
طبق نیاز پروژه، production باید ساده و یکپارچه deploy شود. Dockerfile فعلی هر دو اپ را build و با `turbo start` اجرا می‌کند.

**اگر API تغییر کرد، اول کجا را تغییر بدهم؟**  
اول `docs/API-CONTRACT.md`، بعد `packages/shared`، بعد کد NestJS و Next.js.

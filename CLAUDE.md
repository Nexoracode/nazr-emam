# راهنمای پروژه برای Claude — نذر امام

> این فایل را قبل از هر کاری بخوان. قوانین اینجا روی همه‌ی اپ‌ها و پکیج‌های پروژه اعمال می‌شود.
> اگر بعداً داخل هر اپ `CLAUDE.md` جداگانه اضافه شد، قوانین همان اپ در کنار این فایل رعایت می‌شود.

## پروژه چیست

«نذر امام» یک سامانه فارسی، راست‌به‌چپ و قابل اعتماد برای ثبت، پرداخت، پیگیری و مدیریت نذرهاست.

مسیر اصلی کاربر:
1. مشاهده نوع‌های نذر
2. ثبت درخواست نذر
3. ثبت پرداخت یا رسید
4. دریافت کد رهگیری
5. پیگیری وضعیت درخواست

مسیر اصلی مدیر:
1. مشاهده درخواست‌ها
2. بررسی پرداخت‌ها و رسیدها
3. تغییر وضعیت درخواست‌ها
4. گزارش‌گیری
5. پاسخ به پیام‌ها و تیکت‌ها

## ساختار مونوریپو

```text
apps/web          -> فرانت اصلی با Next.js
apps/api          -> بک‌اند با NestJS
apps/cms          -> مدیریت محتوای قابل ویرایش
packages/shared   -> تایپ‌های مشترک TypeScript و منبع حقیقت قرارداد API
docs              -> workflow، onboarding، قرارداد API و روش review
```

پروژه با `npm workspaces` و `Turborepo` مدیریت می‌شود. ابزار CMS هنوز قطعی نشده و تا قبل از تصمیم کاربر نباید scaffold سنگین برای آن انجام شود.

## قانون شماره ۰ — فقط همان کاری که خواسته شده

- هیچ کاری خارج از درخواست مشخص کاربر انجام نده.
- فیچر اضافه، refactor جانبی، تغییر UX/UI اضافه، تغییر زیرساختی غیرضروری یا cleanup گسترده انجام نده مگر کاربر تأیید کند.
- اگر حین کار باگ یا بهبود مهمی دیدی، اول بپرس: «این را هم انجام بدهم؟»

## قانون شماره ۱ — قرارداد، مقدم بر کد

- شکل هر داده‌ی مشترک بین فرانت و بک، اول در `docs/API-CONTRACT.md` و `packages/shared` تعریف می‌شود، بعد در کد استفاده می‌شود.
- همیشه از تایپ‌های `@nazr-emam/shared` استفاده کن. تایپ تکراری محلی نساز.
- هر route جدید بک‌اند باید همزمان در `docs/API-CONTRACT.md` مستند شود.
- اگر response، request body، status code، error code یا auth یک API تغییر کرد، اول قرارداد و shared را به‌روز کن.

## قانون شماره ۲ — فارسی و RTL

- رابط کاربری فارسی و راست‌به‌چپ است.
- متن‌های UI باید واضح، کوتاه و مناسب کاربر عمومی باشند.
- تا حد ممکن از CSS logical استفاده کن:
  - `padding-inline-start` به‌جای `padding-left`
  - `margin-inline-end` به‌جای `margin-right`
- layout نباید در موبایل یا RTL بشکند.
- فعلاً پروژه چندزبانه نیست؛ بدون درخواست کاربر، سیستم i18n اضافه نکن.

## قانون شماره ۳ — هویت بصری

ظاهر پروژه باید ساده، آرام، قابل اعتماد و مناسب سامانه خدماتی/مذهبی باشد.

- از طراحی شلوغ، تبلیغاتی یا کارت‌های تزئینی بی‌هدف پرهیز کن.
- رنگ‌ها را در CSS variable نگه دار.
- رنگ خام پراکنده در کامپوننت‌ها ننویس.
- دکمه‌های عملیاتی باید واضح باشند؛ فرم‌ها باید خوانا و بدون ابهام باشند.
- اطلاعات حساس کاربر مثل شماره موبایل و کد ملی را بی‌دلیل در UI عمومی نمایش نده.

## قانون شماره ۴ — Docker تک‌کانتینری

- نیاز پروژه این است که NestJS و Next.js در production داخل **یک container** اجرا شوند.
- Dockerfile و `docker-compose.yml` باید این قرارداد را حفظ کنند.
- اگر اسکریپت‌های `build` یا `start` تغییر کرد، مطمئن شو Docker همچنان کار می‌کند.
- پورت‌ها:
  - Frontend: `3000`
  - API: `3001`

## قانون شماره ۵ — متغیرهای محیطی و اطلاعات حساس

- هیچ مقدار حساسی را در کد یا git ننویس:
  - رمز دیتابیس
  - JWT secret
  - کلید پرداخت
  - token سرویس پیامک/ایمیل
- `.env` نباید commit شود.
- `.env.example` باید در git بماند و فقط نام متغیرها و مقدار نمونه امن داشته باشد.
- در API، دسترسی مستقیم پراکنده به `process.env` نساز؛ برای فاز بعد یک config متمرکز و validate شده ایجاد شود.

## قانون شماره ۶ — دیتابیس

دیتابیس هنوز نهایی نشده است. تا وقتی تصمیم قطعی ثبت نشده:

- ORM یا دیتابیس جدید اضافه نکن.
- schema یا migration واقعی نساز.
- اگر نیاز به persistence پیش آمد، اول گزینه‌ها را با کاربر قطعی کن و بعد قرارداد را در docs ثبت کن.

وقتی دیتابیس انتخاب شد، قوانین migration، نام‌گذاری جدول‌ها و config در همین فایل اضافه می‌شود.

## قانون شماره ۷ — کیفیت API

- TypeScript strict رعایت شود.
- از `any` استفاده نکن مگر واقعاً لازم باشد و دلیلش روشن باشد.
- هر endpoint باید ورودی را validate کند.
- خطاها باید با شکل `ApiError` در `@nazr-emam/shared` هماهنگ باشند.
- لیست‌ها باید با شکل `Paginated<T>` برگردند.
- endpointهای admin باید دسترسی admin را enforce کنند.
- داده‌های حساس کاربر نباید در log یا response عمومی لو بروند.

## قانون شماره ۸ — کیفیت Frontend

- فرانت باید از قرارداد `docs/API-CONTRACT.md` و تایپ‌های `@nazr-emam/shared` پیروی کند.
- فرم‌ها باید error state، loading state و حالت موفقیت/شکست قابل فهم داشته باشند.
- صفحه‌ها باید در موبایل و دسکتاپ قابل استفاده باشند.
- متن راهنما زیاد و تبلیغاتی ننویس؛ UI باید خودش قابل فهم باشد.
- برای تجربه‌های اصلی پروژه، مسیر کاربر را کامل بساز: ثبت نذر، پرداخت/رسید، پیگیری وضعیت.

## قانون شماره ۹ — تست بعد از هر تغییر

- بعد از هر تغییر، حداقل بررسی‌های مرتبط را اجرا کن.
- برای تغییرات عمومی پروژه، این دو دستور پیش‌فرض هستند:
  ```bash
  npm run build
  npm run lint
  ```
- اگر تغییر فقط مستندات است و build/lint به کد مربوط نمی‌شود، باز هم وضعیت را شفاف گزارش کن: «تغییر مستنداتی بود، build/lint اجرا شد» یا اگر اجرا نشد، دلیلش را بگو.
- اگر تست یا build شکست خورد، commit نزن و اول مشکل را گزارش یا اصلاح کن.
- در پایان کار، یک چک‌لیست تست به کاربر بده:
  - چه چیزی اجرا شد
  - نتیجه هر مورد چه بود
  - اگر چیزی اجرا نشد، چرا

## قانون شماره ۱۰ — commit بعد از سبز بودن بررسی‌ها

- اگر همه بررسی‌های لازم موفق بود، تغییرات را commit کن.
- commit باید کوچک، منطقی و مرتبط با همان درخواست باشد.
- اگر تست‌ها شکست خوردند، یا Git/محیط اجازه commit نداد، commit نزن و دلیل را شفاف بگو.
- قبل از commit، تغییرات نامرتبط کاربر را revert نکن و داخل commit نیاور مگر برای انجام همین درخواست لازم باشند.

## قانون شماره ۱۱ — گزارش پایان کار

- بعد از اتمام هر کار، یک گزارش کوتاه بنویس.
- گزارش باید شامل این موارد باشد:
  - چه تغییراتی انجام شد
  - چه تست‌هایی اجرا شد و نتیجه چه بود
  - آیا commit زده شد یا نه
  - زمان صرف‌شده تخمینی
  - تخمین زمان انجام دستی همان کار توسط انسان
- تخمین زمان دستی باید واقع‌بینانه باشد و شامل خواندن فایل‌ها، ویرایش، تست و commit شود.

## قانون شماره ۱۲ — گردش کار Git

- مستقیم روی `main` کد نزن.
- برای هر کار branch بساز:
  - `feature/...`
  - `fix/...`
  - `docs/...`
- commitها کوچک و منطقی باشند.
- قبل از پایان کار، build را اجرا کن.

## دستورها

```bash
npm install       # نصب وابستگی‌ها
npm run dev       # اجرای local کل پروژه با Turbo
npm run build     # build همه workspaceها
npm run lint      # بررسی تایپ/لینت workspaceها
```

آدرس‌های local:

```text
Frontend: http://localhost:3000
API:      http://localhost:3001
Health:   http://localhost:3001/health
```

اجرای production با Docker:

```bash
docker compose up --build
```

## چیزهایی که نباید بکنی

- بدون به‌روزرسانی `docs/API-CONTRACT.md` شکل API را عوض نکن.
- تایپ مشترک را داخل `apps/web` یا `apps/api` کپی نکن؛ از `@nazr-emam/shared` استفاده کن.
- بدون نیاز واقعی dependency جدید اضافه نکن.
- بدون تأیید کاربر دیتابیس، ORM، ابزار CMS یا i18n اضافه نکن.
- فایل `.env` یا مقدار حساس را commit نکن.
- Docker تک‌کانتینری را به چند container تبدیل نکن مگر کاربر صریحاً بخواهد.
- متن UI را شلوغ، مبهم یا نامتناسب با فضای پروژه ننویس.
